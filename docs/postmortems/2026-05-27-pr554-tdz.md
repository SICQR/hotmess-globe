# Postmortem — PR #554 TDZ crash on /

**Date:** 2026-05-27
**Impact:** root path (`/`) threw `ReferenceError: Cannot access 'p' before initialization` on first paint. Live users saw a blank app shell. Rolled back same day via #556 (production tree revert) + #557 (main squash revert). Re-shipped clean as #570.
**Duration on production:** ~minutes — rollback was reflexive once the error surfaced.

## What shipped in #554

- `api/feedback.js` (server, irrelevant to client crash)
- `src/components/feedback/PulseFeedbackButton.jsx` — floating chip, **eager** import of `PulseFeedbackSheet` (254 lines)
- `src/components/feedback/PulseFeedbackSheet.jsx` — 254 lines, imports `supabaseClient` and `trackEvent`
- `src/App.jsx` — top-level **eager** import of `PulseFeedbackButton`, mounts it inside `OSArchitecture`
- `src/components/utils/analytics.jsx` — reads `window.__hm_is_beta`
- `src/hooks/useUserContext.js` — writes `window.__hm_is_beta` during render
- `src/pages/admin/AdminFeedbackPage.jsx` (lazy — fine)
- `src/pages/beta/RedeemPage.jsx` (small diff — fine)

## The chain that made the bundle fragile

```
App.jsx (eager)
 └─ PulseFeedbackButton.jsx (eager)
     ├─ useMusicPlayer (top-level hook call → forces MusicPlayerContext into main chunk graph)
     └─ PulseFeedbackSheet.jsx (eager)
         ├─ supabase client (already in main chunk)
         └─ analytics.trackEvent (reads window.__hm_is_beta written by useUserContext)
```

Vite's chunker resolves this into a single bundle with cross-module references. Under certain symbol-collision and module-evaluation orderings, Rollup output emits a `let` declaration *after* its first reference in the chunked file — minified to `p`. First execution of `/` hits it before initialization → TDZ throw → blank app.

The error message `Cannot access 'p' before initialization` is the Rollup-minified form of a circular-init dependency. The variable name is meaningless; the bundler position is everything.

## Why we couldn't catch it locally

- `tsc --noEmit` passes — TypeScript doesn't model module-eval order, only types
- Local `vite dev` works — dev mode evaluates modules on demand, doesn't reproduce the chunked production order
- `vite build` would catch it BUT our 45s sandbox bash cap kills the build mid-way
- Vercel preview WOULD catch it — but the preview rendered "successfully" because preview deploys use a different DNS path and we didn't `?_v=` cache-bust + click around enough to hit the throw

## What #570 changed (and why it worked)

1. **Lazy-loaded the sheet.** `PulseFeedbackSheet` is now imported via `React.lazy(() => import(...))`. This moves it into its own chunk that loads only when the button is tapped — taking it out of the main-chunk dependency graph entirely.
2. **Removed top-level `useMusicPlayer`.** The hook was called inside the button component body unconditionally. #570 dropped it. The bottom-offset calculation was made constant (`currentTrack` was the only use of the hook).
3. **Wrapped in `FeedbackErrorBoundary`.** Even if a future regression throws, the boundary swallows the throw and the app continues to render. Defence in depth.
4. **Feature flag `VITE_FEEDBACK_DISABLED`.** One env-var flip kills the button without a redeploy.

## The pattern to codify

**Rule 1.** Any new component mounted globally in `OSArchitecture` must be:
- Lazy-loaded if it's > ~150 lines OR has > 3 internal imports, AND
- Wrapped in a local ErrorBoundary that returns `null` on throw

**Rule 2.** Top-level hook calls in components mounted at root MUST come from contexts already in the root provider tree. If you have to add a new provider, that's a separate PR.

**Rule 3.** Any client-side global side effect (`window.__hm_is_beta = ...`) belongs in a `useEffect`, not in the body of a hook. Writing globals during render is a footgun: re-renders re-execute the write, SSR/hydration may differ, and other readers can race the writer.

**Rule 4.** "Vercel preview = green" is NOT the same as "user-facing root path loads." Before promoting any PR that touches `OSArchitecture` or `App.jsx` imports, the preview URL must be loaded at `/` AND tapped through at least one route change. The `Lint → Typecheck → Build` check passing means the build completed — not that the app runs.

## What we will NOT do

- Re-enable strict eager imports for all of OSArchitecture (would over-correct; some things genuinely belong eager).
- Block lints on circular-import detection (Vite already does this in build; the issue was not catching the build error in our flow).
- Add Sentry / external error tracking just for this. The boundary + flag pattern is sufficient for V1; we can revisit if class-of-bug repeats.
