# HOTMESS Developer Rules Checklist

## Purpose

Translate governance doctrine into enforceable engineering constraints.

If a rule cannot be implemented:
- in code
- in moderation tooling
- in ranking logic
- in observability
- or in rendering

then it is not operational.

---

# Immutable System Order

Every subsystem must enforce this order:

1. Safety
2. Truth
3. Trust
4. Freshness
5. Readability
6. Relevance
7. Monetization

This hierarchy is mandatory.

---

# Safety Rules

The system must:
- suppress abuse rapidly
- throttle suspicious actors
- prevent harassment amplification
- prevent stalking patterns
- prevent doxxing vectors
- prevent manipulative amplification

Safety overrides ranking.

Always.

---

# Truth Rules

The platform must never:
- fake attendance
- fake density
- fake momentum
- fabricate urgency
- misrepresent venue state
- resurrect stale content as current

Realtime integrity is mandatory.

---

# Trust Rules

Trust is:
- earned
- contextual
- decaying
- reversible

Trust affects:
- ranking weight
- moderation thresholds
- amplification power
- saturation tolerance
- visibility persistence

No actor is permanently trusted.

---

# Ranking Rules

Ranking must optimize for:
- usefulness
- locality
- freshness
- readability
- realtime accuracy

Ranking must NOT optimize for:
- infinite engagement
- rage farming
- spam persistence
- artificial urgency

---

# Feed Rules

Feeds must:
- remain scannable
- avoid repetitive dominance
- preserve local diversity
- reduce clutter under pressure

No single venue, sponsor, or creator may dominate visibility indefinitely.

---

# Boost Rules

Boosts are bounded.

Boosts may:
- increase temporary visibility
- improve discovery probability

Boosts may NOT:
- bypass moderation
- override trust
- exceed saturation caps
- force permanent visibility

Boosts must:
- decay over time
- remain labeled
- respect readability constraints

---

# Saturation Rules

The system must actively resist:
- density collapse
- sponsor monopolies
- creator flooding
- repetitive loops
- district overconcentration

When saturation rises:
- visibility compresses
- repetition penalties increase
- cluster collapse becomes stronger
- boosts weaken

---

# Readability Rules

Readability is operational infrastructure.

The interface must:
- remain breathable
- reduce cognitive overload
- collapse duplicates
- surface strongest signals first

Unreadable density is considered system failure.

---

# Quiet State Rules

Quietness is valid.

The platform must never:
- simulate activity
- fake crowds
- invent momentum
- exaggerate turnout

Honest emptiness builds trust.

---

# Moderation Rules

Moderation systems must support:
- suppression
- freezing
- throttling
- restoration
- appeals
- escalation

All actions must be:
- logged
- reversible where appropriate
- reviewable internally

---

# Observability Rules

Every critical system requires:
- alerting
- thresholding
- anomaly detection
- ownership
- escalation

Critical failures must be detectable in realtime.

---

# Privacy Rules

The system must minimize:
- exact location exposure
- persistent tracking
- movement reconstruction

Default behavior:
- fuzzy locations
- coarse clustering
- privacy-first rendering

---

# Metrics Rules

Operational metrics outrank vanity metrics.

Track:
- trust health
- stale leakage
- readability degradation
- saturation
- correction latency
- abuse velocity

Do NOT optimize primarily for:
- raw impressions
- endless session length
- compulsive engagement

---

# AI Rules

AI systems interacting with HOTMESS must:
- preserve truth
- preserve readability
- avoid hallucinated urgency
- avoid spam amplification
- avoid unsafe inference
- obey governance hierarchy

AI may assist ranking.

AI may NOT bypass governance.

---

# Implementation Doctrine

If a rule cannot be expressed as:
- a threshold
- a score
- a cap
- a moderation state
- a visibility constraint
- a ranking modifier
- an alert

then it is not implementation-ready.
