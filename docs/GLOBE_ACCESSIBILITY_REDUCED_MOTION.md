# Globe Accessibility & Reduced Motion

Purpose: define how the HOTMESS Globe supports:

- reduced motion;
- low stimulation;
- sober navigation;
- cognitive accessibility;
- visual accessibility;
- assistive technologies;
- safer nightlife exploration.

This document exists because the Globe is inherently:

- animated;
- emotional;
- atmospheric;
- dense;
- energetic.

Without accessibility governance the experience can become:

- overwhelming;
- unreadable;
- exclusionary;
- physically uncomfortable;
- unsafe for some users.

Accessibility is NOT:

```txt
an afterthought settings page
```

Accessibility must become:

```txt
a first-class rendering system
```

---

# Core philosophy

## The Globe should adapt to people

Users have different:

- sensory thresholds;
- visual needs;
- cognitive load tolerance;
- nightlife comfort levels;
- emotional states;
- accessibility requirements.

The system should adapt dynamically.

NOT force everyone into:

```txt
maximum stimulation mode
```

---

# Accessibility principles

## Calm is a feature

Low stimulation is NOT:

- boring;
- empty;
- less premium.

Calm interfaces can feel:

- elegant;
- luxurious;
- intentional;
- safe.

---

## Motion should communicate meaning

Animations should:

- guide attention;
- communicate energy;
- explain transitions;
- improve orientation.

Animations should NOT:

- exist everywhere;
- compete constantly;
- pulse infinitely;
- simulate casino mechanics.

---

## Accessibility should affect rendering globally

Accessibility cannot live only in:

```txt
CSS overrides
```

Accessibility must influence:

- render budgets;
- animation systems;
- transitions;
- camera movement;
- density handling;
- interaction timing.

---

# Accessibility modes

## Standard mode

Default Globe experience.

Includes:

- atmospheric motion;
- moderate pulsing;
- cinematic transitions.

---

## Reduced motion mode

Primary accessibility mode.

Purpose:

- reduce vestibular discomfort;
- reduce cognitive overload;
- reduce motion fatigue.

---

# Reduced motion rules

## Disable

- continuous pulsing;
- unnecessary orbit drift;
- excessive particle effects;
- aggressive camera movement;
- rapid stack expansion;
- excessive glow flicker.

---

## Simplify

- transitions;
- easing;
- arcs;
- hover effects;
- atmospheric movement.

---

## Replace with

- fades;
- opacity transitions;
- subtle scale shifts;
- static emphasis.

---

# Low stimulation mode

Different from reduced motion.

Purpose:

- calmer nightlife exploration;
- sober-friendly navigation;
- anxiety reduction;
- cognitive load reduction.

---

# Low stimulation rules

## Reduce

- saturation;
- flashing;
- dense overlays;
- simultaneous animation;
- visual noise.

---

## Prioritize

- readability;
- spacing;
- calm pacing;
- larger touch targets;
- simplified layering.

---

## Environment changes

The Globe should feel:

- quieter;
- calmer;
- softer;
- less chaotic.

NOT:

```txt
broken or disabled
```

---

# Calm navigation mode

Specialized mode for:

- sober users;
- NA/AA support;
- vulnerable users;
- overstimulated users;
- anxious users.

---

# Calm mode priorities

## Navigation

Prioritize:

- safer routes;
- calmer streets;
- lower-density areas;
- trusted venues.

---

## Visual treatment

Use:

- soft gradients;
- lower contrast motion;
- reduced nightlife intensity;
- simplified overlays.

---

## Audio policy

If audio exists:

- lower intensity;
- optional ambient only;
- no autoplay spikes.

---

# Camera accessibility

## Camera movement is high-risk

Aggressive camera movement can cause:

- nausea;
- dizziness;
- disorientation;
- fatigue.

---

# Camera rules

## Reduced motion mode

Camera should:

- move less;
- accelerate slower;
- reduce inertia;
- reduce tilt transitions.

---

## Forbidden

- rapid zoom jumps;
- forced spins;
- constant orbital drift;
- dramatic camera shake.

---

# Globe transitions

Transitions should:

- preserve orientation;
- maintain spatial understanding;
- avoid surprise movement.

---

# Beacon accessibility

## Beacon animations

Beacons should never:

- flash rapidly;
- pulse aggressively;
- create seizure-risk patterns.

---

# Beacon hierarchy

Important signals should use:

- contrast;
- spacing;
- shape;
- layering.

NOT only:

- brightness;
- animation.

---

# Beacon categories and accessibility

## Help / SOS

Must remain:

- visible;
- clear;
- calm;
- unmistakable.

Without:

- panic flashing;
- aggressive animation.

---

## NA / AA / sober support

Must feel:

- calm;
- safe;
- low-pressure.

Visual language should avoid:

- club aesthetics;
- chaotic glow systems.

---

# Text readability

## Labels must remain readable

Avoid:

- tiny typography;
- glow-heavy text;
- low contrast overlays.

---

# Required support

## Adjustable

- text size;
- spacing;
- contrast;
- overlay density.

---

# Color accessibility

## Never rely only on color

Signals should differentiate using:

- shape;
- motion;
- pattern;
- iconography;
- spacing.

---

# Color system requirements

Support:

- high contrast mode;
- color-blind-safe palettes;
- reduced glow intensity.

---

# Audio accessibility

If Globe includes audio layers:

- user-controlled;
- easy mute;
- no autoplay spikes;
- caption support.

---

# Haptics accessibility

Haptics should:

- reinforce important actions;
- remain subtle.

Avoid:

- constant buzzing;
- repeated vibration loops.

---

# Interaction accessibility

## Touch targets

Minimum touch target sizes required.

Dense nightlife maps increase:

- mis-taps;
- frustration;
- accessibility failures.

---

# Required interaction support

## Support:

- keyboard navigation;
- screen readers;
- switch controls;
- list fallback;
- gesture alternatives.

---

# List fallback mode

Critical accessibility system.

Users must NOT be forced into:

```txt
3D map interaction
```

---

# List mode requirements

Must support:

- nearby events;
- nearby venues;
- tickets;
- care spaces;
- sober support;
- saved beacons.

Without requiring:

- spatial interaction.

---

# Screen reader support

The Globe must expose:

- semantic labels;
- logical focus order;
- readable beacon descriptions;
- route announcements.

---

# Example accessible beacon announcement

```txt
Event Beacon. Venue: Dalston Superstore. Starts in 45 minutes. Medium venue activity. Accessible entrance available.
```

---

# Density accessibility

Dense nightlife environments are high-risk for cognitive overload.

Therefore:

- simplify automatically at density thresholds;
- reduce simultaneous overlays;
- prioritize clarity over spectacle.

---

# Notification accessibility

Notifications should:

- remain calm;
- avoid panic language;
- avoid manipulative urgency.

---

# Forbidden patterns

## Forbidden

- flashing countdowns;
- endless pulsing;
- rapid flicker;
- autoplay audio spikes;
- forced motion;
- chaotic transitions;
- casino-style urgency loops.

---

# Accessibility state ownership

Accessibility settings should exist as:

```txt
global rendering state
```

NOT:

```txt
random local component flags
```

---

# Required architecture

Accessibility settings must influence:

- GlobeCameraPolicy;
- BeaconVisibilityEngine;
- LocalMapboxScene;
- EnhancedGlobe3D;
- animation scheduler;
- overlay rendering.

---

# System integration

## AccessibilityController

Single source of truth for:

- reduced motion;
- calm mode;
- high contrast;
- audio preferences;
- overlay density.

---

# Performance relationship

Accessibility often improves performance.

Reduced motion modes should:

- lower GPU load;
- reduce animation loops;
- reduce battery drain.

---

# Acceptance criteria

The Globe succeeds when:

- reduced motion feels intentional and premium;
- low stimulation mode feels elegant rather than stripped-down;
- sober users can navigate comfortably;
- Help/SOS remains visible without panic visuals;
- dense nightlife environments remain readable;
- accessibility settings affect the entire rendering system;
- list fallback fully replaces map interaction when needed;
- users are never trapped in overwhelming motion;
- camera transitions remain comfortable;
- color-blind users can interpret signals safely;
- screen readers can understand Globe surfaces;
- accessibility improves trust rather than feeling like a compliance checkbox.
