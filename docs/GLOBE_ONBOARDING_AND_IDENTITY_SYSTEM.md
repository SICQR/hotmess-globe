# Globe Onboarding And Identity System

Purpose: define onboarding, identity creation, visibility setup, consent configuration, trust scaffolding, and profile architecture across HOTMESS Globe.

This document governs:

- onboarding flows;
- identity setup;
- profile systems;
- visibility permissions;
- consent-first onboarding;
- trust scaffolding;
- pseudonymity;
- safety gating;
- accessibility onboarding;
- emotional onboarding design.

The onboarding system should feel:

```txt
welcoming, calm, culturally aware, and low-pressure
```

not:

```txt
extractive, overwhelming, hyper-sexualised, or psychologically manipulative
```

---

# Core philosophy

Onboarding exists to:

- establish safety;
- establish trust;
- establish consent;
- establish emotional tone;
- help users orient;
- help users configure boundaries;
- help users control visibility.

Onboarding does NOT exist to:

- maximise data extraction;
- maximise virality;
- maximise profile completion pressure;
- maximise addictive engagement;
- force identity exposure.

---

# Core identity philosophy

Identity on HOTMESS Globe should support:

- pseudonymity;
- layered visibility;
- selective disclosure;
- contextual identity;
- privacy-safe exploration.

The system must support users who are:

- publicly out;
- partially out;
- anonymous;
- closeted;
- questioning;
- recovering;
- socially cautious.

---

# Identity is layered

Identity should NOT be binary.

Users should control:

| Layer | Visibility |
|---|---|
| display name | configurable |
| alias | configurable |
| pronouns | optional |
| age band | approximate |
| face visibility | optional |
| body visibility | optional |
| district presence | configurable |
| online status | configurable |
| exact location | never public |
| social links | optional |
| recovery visibility | protected |

---

Recommended identity ladder:

```txt
anonymous

→ contextual profile
→ trusted profile
```

Critically:

```txt
identity reveal is separate from location reveal
```

# Recommended onboarding flow

## Step 1 — Welcome

Introduce:

- Globe philosophy;
- care-first approach;
- consent principles;
- visibility control;
- non-gamified culture.

Example:

```txt
You control how visible you are.
```

---

## Step 2 — Age gating

Requirements:

- 18+ mandatory;
- adult consent acknowledgement;
- region-compliant age confirmation.

The system must:

- avoid fetishised age language;
- avoid exploitative onboarding.

---

## Step 3 — Identity setup

Required:

- display name or alias;
- approximate age band;
- optional pronouns;
- optional photo.

Optional identity:

- tribe/vibe;
- nightlife preference;
- accessibility needs;
- sobriety preference;
- radio/community interests.

All optional.

---

## Step 4 — Visibility configuration

Users configure:

- public visibility;
- district visibility;
- Chill discoverability;
- Ghosted chat permissions;
- message permissions;
- proximity visibility;
- map visibility;
- venue visibility.

Defaults should favour:

```txt
privacy and calmness
```

---

## Step 5 — Consent system

Users configure:

- who may message;
- who may Boo;
- visibility radius;
- trusted contacts;
- meetup escalation;
- route-share permissions.

Consent should remain:

- reversible;
- transparent;
- granular.

---

## Step 6 — Safety systems

Introduce:

- trusted contacts;
- SOS systems;
- reporting;
- ghosting;
- block systems;
- privacy controls.

The system should communicate:

```txt
safety without fear theatre
```

---

## Step 7 — Accessibility + stimulation

Users may configure:

- reduced motion;
- reduced stimulation;
- quiet mode;
- nightlife suppression;
- low-density mode;
- digest-first notifications.

This should happen:

```txt
before realtime overwhelm occurs
```

---

## Step 8 — District orientation

Introduce:

- local districts;
- warmth system;
- beacon types;
- care infrastructure;
- radio integration;
- local vibe.

The Globe should feel:

```txt
cultural rather than transactional
```

# Trust scaffolding

Trust should grow gradually.

Examples:

| State | Capability |
|---|---|
| new | limited visibility |
| contextual | Chill + event interaction |
| mutual | expanded messaging |
| trusted | optional deeper visibility |

Trust should NEVER:

- become public status hierarchy;
- become clout infrastructure.

---

# Boo governance

Boo should remain:

```txt
lightweight social acknowledgement
```

NOT:

- desirability scoring;
- public validation economy;
- ranking metric.

# Reduced stimulation onboarding

Reduced stimulation mode should be:

- visible early;
- easy to enable;
- explained clearly.

- ADHD;
- anxiety;

- burnout;
- recovery.

# Accessibility

Onboarding must support:

- screen readers;
- keyboard navigation;
- reduced motion;
- reduced stimulation;
- high contrast;
- low cognitive load.

The onboarding should remain understandable without:

- animation;
- gesture-only flows;
- colour-only signalling.

---

# Anti-dark-pattern policy

The onboarding must NEVER:

- force contacts upload;
- force photos;
- force GPS precision;
- force public visibility;
- guilt incomplete profiles;
- manipulate emotional vulnerability.

Forbidden:

```txt
Complete your profile to get more attention
```

---

# Sponsored onboarding policy

Sponsored onboarding:

- must remain separate;
- must never mimic care systems;
- must never override consent flows.

No onboarding ads should:

- simulate community urgency;
- imitate safety prompts.

- care systems remain protected;

# Acceptance criteria

The onboarding and identity system succeeds when:

- users feel safe before they feel exposed;
- identity remains flexible and contextual;
- privacy controls are understandable;
- visibility is granular and reversible;
- onboarding feels culturally grounded rather than corporate;
- reduced stimulation users feel accommodated immediately;
- consent feels embedded rather than performative;
- care systems remain protected;
- exact location never becomes public by default;
- the Globe feels welcoming without becoming emotionally manipulative.
