# Trust System Spec

## Purpose

The Trust System defines how credibility is earned, lost, applied, and enforced across HOTMESS.

It exists to:
- protect map integrity
- reduce abuse and spam
- support truthful ranking
- shape signal persistence
- determine amplification rights
- keep the ecosystem legible under pressure

Trust is operational permission.

---

# Core Principle

Trust measures how much the system can safely:
- believe
- amplify
- persist
- rank
- expose

High trust means:
- more reliability
- more ranking stability
- softer throttling
- stronger corroboration weight

Low trust means:
- more skepticism
- harsher suppression
- reduced persistence
- stronger cooldowns

---

# Trust Layers

1. User Trust
2. Venue Trust
3. Event Trust
4. Promoter Trust
5. District Trust Context

Trust is contextual.

No single number should fully define credibility.

---

# Core Trust Rules

- Trust is earned
- Trust decays
- Trust is reversible
- Trust is contextual
- Trust affects visibility
- Trust affects amplification
- Trust cannot bypass safety
- Trust cannot override freshness
- Trust cannot be purchased

---

# Positive Inputs

Trust grows through:
- accurate updates
- corroborated claims
- verification
- low dispute rate
- low abuse rate
- consistency over time
- moderation stability

---

# Negative Inputs

Trust decreases through:
- spam
- harassment
- misinformation
- contradiction
- manipulation
- repeated disputes
- abuse reports
- saturation abuse

---

# Verification States

## Unverified
Lowest authority.

## Soft Verified
Moderate confidence.

## Verified
Strong operational confidence.

## High-Trust
Strongest trust band.

## Restricted
Limited visibility and permissions.

Verification must remain revocable.

---

# Trust Effects

Trust affects:
- ranking
- persistence
- decay speed
- suppression thresholds
- boost efficiency
- cooldowns
- district influence

High trust:
- improves stability
- improves persistence
- reduces decay slightly

Low trust:
- accelerates suppression
- accelerates decay
- reduces amplification

---

# Trust Decay

Trust decays when:
- inactivity persists
- stale behavior accumulates
- reports rise
- contradictions emerge
- manipulation appears

Different actors decay differently:
- users decay faster
- venues decay slower
- promoters decay aggressively under abuse

---

# Freeze State

Trust may freeze during:
- moderation review
- conflicting evidence
- safety investigation
- manipulation suspicion

Frozen trust:
- halts amplification
- limits visibility
- requires review

---

# Recovery

Recovery requires:
- sustained good behavior
- accurate updates
- low dispute rate
- successful verification
- moderation approval where required

Recovery should be slower than collapse.

---

# Trust and Ranking

Trust modifies:
- freshness persistence
- momentum confidence
- visibility duration
- boost efficiency

Trust never resurrects stale data.

---

# Trust and Boosts

High trust:
- improves boost efficiency slightly
- delays saturation slightly

Low trust:
- reduces boost value
- increases saturation penalties
- increases cooldowns

Boosts never create trust.

---

# Trust and Moderation

Moderation actions directly affect trust.

Examples:
- abuse findings
- spam patterns
- impersonation
- misinformation
- manipulation campaigns

All trust mutations should:
- be logged
- be reviewable
- support appeals

---

# Contextual Trust Categories

Recommended categories:
- location confidence
- timing confidence
- occupancy confidence
- event accuracy
- abuse risk
- identity consistency

Trust should vary by claim type.

---

# Appeals

Appeals must support:
- correction
- review
- recovery
- verification restoration

Appeals should not instantly reset authority.

---

# Immutable Trust Invariants

- Trust is earned
- Trust decays
- Trust is contextual
- Trust is reversible
- Trust affects visibility
- Trust affects amplification
- Trust cannot be bought
- Trust never overrides safety
- Trust never overrides freshness
