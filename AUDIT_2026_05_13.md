**`order_items` has zero policies** — this is the line-items table joined onto `orders`. If the front-end ever reads it via the anon client (not service role), it will silently return empty rows. Recommend a `SELECT … USING (EXISTS (… orders.user_id = auth.uid()))` policy.

### Policy summaries on critical "destination" tables

- **`profiles`** — `Public profiles are viewable by everyone` (cmd=SELECT, qual=`true`) plus `profiles_read_all_auth` (SELECT, true) plus `profiles_select_own`. **Two duplicate "read-all" policies.** Plus `operator_no_individual_location` SELECT qual=`false` — looks intended to block operator-tier viewers from individual rows but, given how Postgres RLS unions SELECT policies, **it has zero effect** (any matching row from the other two policies wins). Cleanup needed.
- **`taps`** — five policies, two of which (`users can view taps they sent or received` and `taps_select_own`) overlap. `taps_select_own` correctly handles both `from_user_id`/`to_user_id` AND `tapper_email`/`tapped_email` cases (because of the dual-id columns). OK but verbose.
- **`messages`** (canonical chat table) — `msg_select_if_member` / `msg_insert_if_member` / `msg_update_own` / `msg_delete_own` plus duplicate `members_can_*` + `sender_can_*`. All gated correctly via `conversation_members`. ✓
- **`chat_messages`** (deprecated) — four overlapping policies, all email-based (`auth.jwt() ->> 'email'`). **Will fail** for any user who signs in with OTP/passwordless and has email cleared in JWT. Slated for DROP 2026-05-21.
- **`profile_photos`** — owner can ALL, others can SELECT only if `moderation_status='approved'`. ✓ Clean.
- **`ghosted_albums` / `ghosted_album_photos` / `ghosted_album_shares`** — owner can ALL; recipient SELECT gated by `has_active_album_share()` and `has_xxx_access()` SQL functions. ✓ Clean. This is the Grindr-style lockbox flow — looking good.
- **`right_now_status`** — **this is a VIEW, not a table.** Views don't have RLS — they inherit from their underlying tables. Confirmed in `information_schema.tables`. Phil's CLAUDE.md notes this — Code that writes "to right_now_status" must hit the TABLE that backs it (`user_intents` + `user_presence_locations` per recent commits).
- **`beacons`** — six policies. Two duplicate SELECT policies (`Anyone can view active beacons` vs `beacons_read_all`). UPDATE/DELETE include admin role override. ✓
- **`events`** — only ONE policy: `Creators can manage own events` (cmd=ALL, qual=`auth.uid() = creator_id`). **No SELECT for anyone else.** Events table has 0 rows in prod, but the moment one is created, only its creator can read it. Bug. Should mirror beacons.
- **`blocks`** — owner manages own; blocked party can see they're blocked. ✓
- **`reports`** — reporter inserts + selects own. ✓ No admin read policy — admins must use service role.
- **`market_listings`** — multiple INSERT policies (Users can create listings + Users can create their own listings + listing_insert_seller — three different definitions). SELECT has a duplicate `Anyone can see listings` (qual=`true`) vs `Anyone can view active listings` (qual=`status IN ('active','live')`) — the former wins, **so soft-deleted listings are still publicly visible.** Cleanup needed.
- **`market_sellers`** — duplicate INSERT policies (`Users can create their own seller profile` + `Users can insert their own seller profile` + `market_sellers_insert_auth`). All open-INSERT (no qual). Fine but messy.
- **`profile_views`** — **ZERO policies registered** despite RLS being enabled. Re-confirmed in pg_policies query — the table sits inside the "RLS-enabled, no policies" gap. The frontend (L2ProfileSheet) DOES insert here on every profile view. **This will silently fail in production** — only service-role inserts succeed. Bug.
- **`feature_flags`** — admin write, authenticated read. ✓

### Highest-severity advisors

1. **`spatial_ref_sys` RLS disabled** — known PostGIS quirk. Optional remediation but low practical risk.
2. **Function search_path mutable** — multiple SECURITY DEFINER functions without explicit `SET search_path`. Medium risk.
3. **SECURITY DEFINER views** — some views run with definer privileges that bypass RLS in unexpected ways. Includes `presence`, `beacons`, `right_now_status`. Worth a deliberate audit pass.
4. **OTP expiry** (auth config, not RLS) — still flagged in CLAUDE.md as needing 3600 max.
5. **Leaked-password protection** — still flagged off.
6. **Postgres minor version** — upgrade available.
